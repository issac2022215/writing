# 小红书爆款文案生成器改进设计

## 背景

当前项目是一个基于 Streamlit 的单文件应用（`main.py`，约 115 行），核心功能是通过 OpenAI/DeepSeek 大模型生成小红书风格文案。用户希望综合提升，主要痛点：

1. **API Key 每次手动输入**（但明确不需要持久化 Key）
2. **生成后文案无法管理**（不能保存、搜索、导出）

## 设计原则

- **安全优先**：API Key 不持久化，仅当前会话有效
- **模块化**：将单文件拆分为职责清晰的模块，便于维护和扩展
- **渐进增强**：保持现有功能完整可用，在此基础上叠加新功能

### 模块职责

| 模块 | 职责 | 对外接口 |
|------|------|---------|
| `config.py` | 模型列表、默认参数、选项映射 | 常量字典和列表 |
| `prompts.py` | 根据参数动态拼装 system prompt | `build_system_prompt(style, word_count, emoji_level) -> str` |
| `generator.py` | 调用 OpenAI/DeepSeek API，流式返回 | `generate_stream(base_url, api_key, model, messages) -> Generator[str]` |
| `storage.py` | 文案的增删查 | `save()`, `search()`, `list_all()`, `delete()` |
| `ui/*.py` | Streamlit 组件，纯 UI 渲染 | 每个文件一个 `render_*()` 函数 |

## 2. 功能

### 2.1 生成参数面板（侧边栏）

用户在输入关键词之前，可调节三个参数，参数值动态注入 system prompt：

| 参数 | 选项 | 默认值 | 说明 |
|------|------|--------|------|
| 文案风格 | 幽默 / 严肃 / 温馨 / 轻松 / 真诚 / 热情 / 鼓励 | 轻松 | 控制行文口吻 |
| 字数范围 | 短（200字）/ 中（500字） / 长（800字） | 中 | 控制正文字数上限 |
| Emoji 浓度 | 少 / 适中 / 多 | 适中 | 控制 emoji 使用频率 |

### 2.2 生成结果操作

每条生成的文案下方提供三个操作按钮：

- **📋 复制**：一键复制完整文案（标题+正文）到剪贴板
- **📥 下载**：导出为 `.md` 文件下载
- **⭐ 收藏**：标记为重要，置顶显示

### 2.3 历史记录（侧边栏底部）

- **会话内快捷回看**：本次会话生成的关键词以标签形式显示在输入框下方，点击快速回填
- **本地持久化**：SQLite 存储所有历史文案，关闭应用后再次打开仍可查看
- **搜索过滤**：侧边栏历史列表顶部有搜索框，按关键词筛选
- **点击回看**：点击历史记录在结果区展示完整文案
- **单条删除**：每条历史记录旁有删除按钮

## 3. 提示词设计

### 模板化方案

System prompt 从硬编码字符串改为参数化模板。核心创作规则（二极管标题法、爆款关键词等）保持不变，但根据用户参数动态调整部分指令：

- `{style}` 注入写作风格要求
- `{word_count}` 控制正文字数上限
- `{emoji_level}` 控制 emoji 使用频率

`prompts.py` 中定义风格和 emoji 浓度到具体指令的映射，`build_system_prompt()` 函数负责拼装最终 prompt。

### 扩展性

后续可以轻松添加新的模板变体：
- 不同的平台风格（知乎、公众号、微博）
- 不同的文案类型（探店、测评、教程）
- 自定义 prompt（高级用户手动编辑）

## 4. 存储设计

### 数据库

使用 Python 标准库 `sqlite3`，无需额外依赖。数据库文件存储在应用目录下：

```sql
CREATE TABLE IF NOT EXISTS history (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    keyword TEXT NOT NULL,
    titles TEXT NOT NULL,        -- JSON 数组，存储标题列表
    content TEXT NOT NULL,       -- 正文
    style TEXT,                  -- 生成时的风格参数
    word_count TEXT,             -- 生成时的字数参数
    emoji_level TEXT,            -- 生成时的 emoji 浓度参数
    model_name TEXT,             -- 使用的模型
    is_favorite INTEGER DEFAULT 0,
    created_at TEXT DEFAULT (datetime('now', 'localtime'))
);

CREATE INDEX IF NOT EXISTS idx_keyword ON history(keyword);
CREATE INDEX IF NOT EXISTS idx_created_at ON history(created_at DESC);
```

### 操作接口

```python
def save(keyword, titles, content, style, word_count, emoji_level, model_name) -> int
def search(keyword: str = "", limit: int = 50) -> list[dict]
def get_by_id(id: int) -> dict | None
def delete(id: int) -> bool
def toggle_favorite(id: int) -> bool
```

## 5. 错误处理

将单层 `try/except Exception` 替换为分层错误处理：

| 场景 | 检测方式 | 用户提示 |
|------|---------|---------|
| API Key 为空 | 前置检查 | "请提供访问大模型需要的 API Key" |
| API Key 无效 | HTTP 401 | "API Key 校验失败，请检查是否正确填写" |
| 网络超时 | `APITimeoutError` | "请求超时，请检查网络连接后重试" |
| 余额不足 | HTTP 429 / billing error | "API 额度可能已用完，请检查账户余额" |
| 模型不可用 | HTTP 404 / model not found | "所选模型当前不可用，请尝试切换其他模型" |
| 流式中断 | `APIError` / 网络断开 | "生成过程中连接中断，已生成的内容保留在下方" |
| 其他未知错误 | 兜底 catch | 显示具体错误信息，而非笼统提示 |

## 7. 实施顺序

按优先级分阶段：

| 阶段 | 内容 | 预估 |
|------|------|------|
| 1 | 代码拆分：将 `main.py` 拆为 `config.py`、`prompts.py`、`generator.py`、`storage.py`，功能不变 | 先做 |
| 2 | UI 重构：拆出 `ui/` 子模块，加入生成参数面板和结果操作按钮（复制/下载） | 接着做 |
| 3 | 历史记录：SQLite 存储 + 搜索 + 回看 + 删除 | 最后做 |
| 4 | 错误处理优化：分层 error handling | 穿插在各阶段 |

每个阶段完成后保持应用可运行，不做一次大爆炸式重构。
